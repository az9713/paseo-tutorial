● Root-caused. It's not a misconfiguration — it's how import is designed to
  work. You're missing one assumption about what "import" means, not a setting.

  The verdict

  Paseo does not live-watch the provider session files on disk. Importing a
  Codex/Claude session takes a one-time snapshot and then runs Paseo's own
  provider process. Anything you type into a separate claude/codex terminal on
  the laptop is a different process writing to disk that Paseo neither tails nor
  merges. Live updates in Paseo only reflect work that flows through Paseo.

  The mental-model gap

  You're treating "import" as "mirror my live terminal session." In the code it
  actually means "take over / resume this conversation as a Paseo-controlled
  agent." After import there are effectively two independent conversations:

  ┌──────────────────────────┬────────────────────────┬─────────────────────┐
  │      Where you type      │       What it is       │ Does Paseo see it   │
  │                          │                        │        live?        │
  ├──────────────────────────┼────────────────────────┼─────────────────────┤
  │ In a Paseo client        │ Paseo's own provider   │ ✅ Yes — streamed   │
  │ (phone/web/CLI)          │ process                │ to all clients      │
  ├──────────────────────────┼────────────────────────┼─────────────────────┤
  │ In a separate            │ A different process on │ ❌ No — not         │
  │ claude/codex terminal    │  the same machine      │ watched, not merged │
  └──────────────────────────┴────────────────────────┴─────────────────────┘

  The proof from the codebase

  Three facts, each from the source:

  1. Import = snapshot + resume, not a live feed. importProviderSession calls
  resumeAgentFromPersistence (starts Paseo's own process) then
  hydrateTimelineFromProvider — which reads provider history exactly once
  (packages/server/src/server/agent/import-sessions.ts:172-181).

  2. There is no file watcher. Nothing in the daemon does fs.watch/tail on
  ~/.claude/projects or ~/.codex/sessions. Updates come from the provider
  process Paseo controls — not from observing the file.

  3. The code says external edits need a manual reload — in a comment, verbatim
  (agent-manager.ts:874-879):

  ▎ "When rehydrateFromDisk is set, the timeline is wiped so a new epoch is
  ▎ minted and provider history is re-streamed — this is what the user-facing
  ▎ 'Reload agent' action wants when the on-disk session was mutated outside
  ▎ Paseo."

  And that "Reload agent" path (session.ts:3343 → reloadAgentSession(agentId, {
  rehydrateFromDisk: true })) is the only mechanism that re-reads the disk after
  an external change. It's a one-shot re-read, not continuous sync.

  ★ Insight ─────────────────────────────────────
  This is a direct consequence of the architecture in your new docs: the daemon
  reads provider session files, it doesn't subscribe to them. Reading-on-demand
  is cheap and lock-free (great for the "list recent sessions" feature);
  live-tailing would mean watching every provider's log format for external
  writes and reconciling two processes editing the same conversation — which is
  exactly the divergence Paseo avoids by having you drive the agent through it.
  ─────────────────────────────────────────────────

  What to do

  Option A — drive the conversation from Paseo (recommended). After importing,
  send your follow-ups from the Paseo app/CLI, not from the laptop terminal.
  Then every device stays in sync automatically. Treat import as "I'm moving
  this conversation into Paseo now."

  Option B — keep using the terminal, then manually pull changes. If you want to
  keep typing in the laptop claude/codex terminal, you must explicitly re-read
  the disk to see those messages in Paseo:

  paseo agent reload <agent-id>

  (or the "Reload agent" action in the app). This re-hydrates the imported
  agent's timeline from the current on-disk state. It is manual and one-shot —
  you re-run it each time you want to catch up. There is no toggle for automatic
  live external sync — so you're not missing a setting that would enable it.

  One caveat to be aware of

  Running both at once — the live terminal session and the imported Paseo agent
  — on the same conversation is asking for trouble. They're two processes; once
  you send a message through Paseo, its agent and your terminal session can
  diverge into separate branches, and reload only ever reflects whatever is
  currently on disk for that session handle. Pick one driver per conversation.
